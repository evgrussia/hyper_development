## Phase 1: Авторизация (Telegram)

**Цель:** вход через Telegram Login Widget (web) и через Telegram WebApp (Mini App).

### 1.1 Авторизация через Telegram Widget (web)

| # | Шаг | Детали |
|---|-----|--------|
| 1.1.1 | Bot Token и проверка данных | В BotFather получить токен; на backend хранить `TELEGRAM_BOT_TOKEN`; проверять `hash` из callback по алгоритму Telegram (HMAC-SHA256 с secret key = SHA256(bot_token)) |
| 1.1.2 | Модель/профиль пользователя по Telegram | Поле `telegram_id` (bigint, unique), `username`, `first_name`, `last_name`, `photo_url`; связь с `User` (OneToOne или кастомная user model) |
| 1.1.3 | API endpoint | `POST /api/auth/telegram/` — тело: `id`, `first_name`, `last_name`, `username`, `photo_url`, `auth_date`, `hash`. Валидация hash → создание/обновление пользователя → выдача JWT (access + refresh) |
| 1.1.4 | Frontend: виджет | Подключить Telegram Login Widget (script от Telegram), после успешного callback отправить данные на `POST /api/auth/telegram/` и сохранить токены (httpOnly cookie или localStorage), редирект в ЛК |

### 1.2 Авторизация для Telegram WebApp

| # | Шаг | Детали |
|---|-----|--------|
| 1.2.1 | Получение initData | В WebApp использовать `window.Telegram.WebApp.initData` (строка с user, auth_date, hash и т.д.) |
| 1.2.2 | Верификация initData на backend | Проверка подписи: разбор query string, проверка `hash` (HMAC-SHA256 с ключом = HMAC-SHA256("WebAppData", bot_token)); проверка `auth_date` не старше N минут (например 24 ч) |
| 1.2.3 | API для WebApp | `POST /api/auth/telegram/webapp/` — тело: `init_data` (raw string). Верификация → user by telegram_id → JWT. Либо отдельный short-lived token для WebApp |
| 1.2.4 | Frontend WebApp | При запуске Mini App отправить `init_data` на backend, получить токен, использовать его в заголовке для всех API |

**Результат Phase 1:** пользователь может войти с web-сайта через виджет и из Telegram Mini App; backend выдаёт JWT и хранит профиль по `telegram_id`.

## Phase 4: Реферальная и партнёрская системы

**Цель:** реферальные ссылки, учёт привлечённых, партнёрская программа и её аналитика.

### 4.1 Реферальная система

| # | Шаг | Детали |
|---|-----|--------|
| 4.1.1 | Модели | `ReferralLink`: user (FK), code (unique, например uuid или короткий код), created_at; `Referral`: referrer (FK User), referred (FK User), created_at; при регистрации/первой оплате записывать referred |
| 4.1.2 | Генерация ссылки | В ЛК: "Моя реферальная ссылка" — генерировать код и ссылку вида `https://t.me/bot?start=ref_CODE` или `https://site.com/?ref=CODE` |
| 4.1.3 | Учёт рефералов | При регистрации по ref или при первом действии (например оплата) создавать `Referral` и привязывать к referrer; показывать в ЛК список привлечённых и уровень (если многоуровневая система) |

### 4.2 Партнёрская программа

| # | Шаг | Детали |
|---|-----|--------|
| 4.2.1 | Модели | `Partner`: user (OneToOne), status (pending/approved/blocked), commission_percent, level; `PartnerPayout`: partner, amount, order/target, status, paid_at |
| 4.2.2 | Логика начисления | При оплате заказа (Phase 5): вычислять комиссию партнёра по реферальной цепочке; создавать записи `PartnerPayout` (pending), затем по регламенту переводить в paid |
| 4.2.3 | API для партнёра | `GET /api/partners/me/` — статистика (клики, регистрации, продажи, начисления), `GET /api/partners/me/payouts/` |

### 4.3 Аналитика партнёрской программы

| # | Шаг | Детали |
|---|-----|--------|
| 4.3.1 | Метрики | Хранить события: переход по ref, регистрация по ref, первая оплата по ref; агрегаты по партнёру: количество рефералов, конверсия, сумма начислений |
| 4.3.2 | API и отчёты | Дашборд в ЛК партнёра: графики (по дням/неделям), таблица выплат; при необходимости экспорт CSV |

**Результат Phase 4:** реферальные ссылки работают, партнёры видят статистику и выплаты; аналитика партнёрки доступна в ЛК и для админки.

## Phase 5: Платежи (TON, ЮKassa)

**Цель:** приём оплаты в TON и через ЮKassa; связь с заказами и партнёрскими начислениями.

### 5.1 ЮKassa

| # | Шаг | Детали |
|---|-----|--------|
| 5.1.1 | Модели | `Order`: user, items (JSON или FK к позициям), total, currency, status (created/pending/paid/failed/refunded), payment_system (yookassa/ton), external_id (id в платёжной системе), created_at |
| 5.1.2 | Интеграция | Использовать официальный SDK или `requests`: создание платежа (amount, return_url, description, metadata: order_id, user_id); сохранение `payment_id` и `confirmation_url` в Order |
| 5.1.3 | Webhook | `POST /api/payments/yookassa/webhook/` — проверка подписи/секрета, обновление Order по статусу; при success — начисление партнёру (Phase 4), выдача доступа к товару/услуге при необходимости |
| 5.1.4 | Frontend | После создания заказа редирект на `confirmation_url`; после возврата с ЮKassa — страница "Оплата прошла" / "Ожидаем оплату" с опросом статуса по API при необходимости |

### 5.2 TON

| # | Шаг | Детали |
|---|-----|--------|
| 5.2.1 | Выбор подхода | TonConnect (кошелёк в браузере) или приём на единый кошелёк с уникальной суммой/мемо для идентификации заказа |
| 5.2.2 | Модель | Тот же `Order` с `payment_system='ton'`; хранить адрес кошелька, ожидаемую сумму в nanoTON, memo/comment для идентификации |
| 5.2.3 | Подтверждение оплаты | Опция 1: TonConnect + подпись транзакции от клиента и проверка на backend. Опция 2: сканирование блокчейна по адресу и memo (cron или очередь) — при поступлении суммы обновлять Order и начислять партнёру |
| 5.2.4 | API | `POST /api/orders/` создаёт заказ и возвращает данные для оплаты (address, amount, memo); фронт показывает инструкцию или открывает TonConnect |

**Результат Phase 5:** заказы создаются, оплата возможна через ЮKassa и TON; статусы обновляются, партнёрские начисления считаются.

## Phase 6: Бизнес-логика (плагинная архитектура)

**Цель:** разная логика для разных проектов без переписывания ядра.

### Шаги

| # | Шаг | Детали |
|---|-----|--------|
| 6.1 | Концепция плагинов | Отдельное Django-приложение `business_logic` или папка `plugins/` с подключаемыми модулями; каждый проект — свой набор плагинов (один проект: подписки, другой: маркетплейс и т.д.) |
| 6.2 | Интерфейсы/контракты | Абстрактные классы или протоколы: например `OrderFulfillmentHandler` (при оплате заказа — выдать доступ, отправить товар и т.д.), `UserOnboardingSteps`, `EligibilityChecker` для партнёрских начислений |
| 6.3 | Регистрация плагинов | В настройках перечисление активных плагинов; при старте приложения регистрация обработчиков в единый диспетчер (сигналы Django или вызовы из views/services) |
| 6.4 | Примеры | Плагин "Подписка": после оплаты создаёт Subscription до даты; плагин "Цифровой товар": выдаёт ссылку на скачивание; плагин "Курс": добавляет запись в группу доступа к курсу |
| 6.5 | Конфигурация по проекту | Переменные окружения или настройки в БД/админке: какой плагин активен, параметры (id продукта подписки, id курса и т.д.) |

**Результат Phase 6:** одна кодовая база поддерживает разные сценарии за счёт плагинов; добавление нового MVP = новый плагин + конфиг.

## Phase 7: AI-агенты (системный, персональный, RAG)

**Цель:** системный LangChain-агент с общей RAG; персональный агент с персональной RAG; базы знаний пополняются из ЛК.

### 7.1 Общая инфраструктура RAG

| # | Шаг | Детали |
|---|-----|--------|
| 7.1.1 | Векторное хранилище | Выбор: Chroma, Qdrant, Weaviate, pgvector (PostgreSQL). Для универсальности — pgvector или отдельный сервис (Chroma/Qdrant) в Docker |
| 7.1.2 | Эмбеддинги | LangChain интеграция: OpenAI Embeddings или open-source (sentence-transformers, локально). Модель в настройках |
| 7.1.3 | Индексация документов | Па pipeline: загрузка файла/текста → чанкинг (RecursiveCharacterTextSplitter или аналог) → эмбеддинги → сохранение в векторный store с метаданными (source, project_id, user_id для персональной базы) |

### 7.2 Системный AI-агент (LangChain)

| # | Шаг | Детали |
|---|-----|--------|
| 7.2.1 | Назначение | Управление бизнес-логикой (ответы по правилам, продуктам, процессам), исследования (web search tool), рекомендации на основе общей базы знаний |
| 7.2.2 | Tools | RAG retriever (общая коллекция), web search (SerpAPI, Tavily или другой), калькулятор/внутренние API (статус заказа, список продуктов) при необходимости |
| 7.2.3 | Граф/агент | LangGraph: узел "retriever" → узел "llm" → tool calls → ответ; state с messages и контекстом из RAG |
| 7.2.4 | API | `POST /api/ai/system/chat/` — сообщение пользователя; ответ stream или JSON; аутентификация по JWT |
| 7.2.5 | Пополнение общей базы из ЛК | В ЛК раздел "База знаний (системная)": загрузка файлов (PDF, TXT, MD); backend обрабатывает и индексирует в общую RAG-коллекцию; права только для админов или доверенных ролей |

### 7.3 Персональный AI-агент

| # | Шаг | Детали |
|---|-----|--------|
| 7.3.1 | Отдельная RAG на пользователя | Векторная коллекция с `metadata.user_id = current_user.id`; при запросе retriever фильтрует по user_id |
| 7.3.2 | Назначение | Персональные рекомендации, ответы на основе только своих документов и данных (мои заказы, мои заметки) |
| 7.3.3 | Tools | Персональный RAG retriever; опционально инструменты "мои заказы", "мои статьи" через внутренние API |
| 7.3.4 | API | `POST /api/ai/personal/chat/` — контекст только своего пользователя |
| 7.3.5 | Пополнение из ЛК | В ЛК: "Моя база знаний" — загрузка файлов; индексация в персональную коллекцию (user_id в metadata) |

### 7.4 Безопасность и лимиты

| # | Шаг | Детали |
|---|-----|--------|
| 7.4.1 | Изоляция данных | Системный агент — только общая коллекция; персональный — только свои документы и свои инструменты |
| 7.4.2 | Лимиты | Rate limit по пользователю; лимит размера/количества документов в персональной RAG (настраиваемо в админке) |

**Результат Phase 7:** два агента (системный и персональный) работают через API; обе RAG пополняются из ЛК; исследования и рекомендации завязаны на LangChain/LangGraph.

## Phase 8: Уведомления и AI-посты в Telegram

**Цель:** отправка уведомлений пользователям в Telegram; AI-агент для генерации постов в канал по сценарию и расписанию.

### 8.1 Система уведомлений в Telegram

| # | Шаг | Детали |
|---|-----|--------|
| 8.1.1 | Модель | `Notification`: user (FK), type (order_paid, referral_bonus, system и т.д.), title, body, sent_at, read_at; или только лог отправок в Telegram (telegram_id, text, sent_at) |
| 8.1.2 | Отправка | Использовать Bot API: `sendMessage(chat_id=user.telegram_id, text=...)`; при отсутствии telegram_id — пропуск или fallback (email позже) |
| 8.1.3 | События | Сигналы Django или вызовы из сервисов: после оплаты заказа, после регистрации реферала, ручная рассылка из админки; очередь (Celery) для массовых уведомлений |
| 8.1.4 | Шаблоны | Шаблоны сообщений в БД или в коде (тип события → шаблон с подстановкой имени, суммы и т.д.) |

### 8.2 AI-агент для постов в канал

| # | Шаг | Детали |
|---|-----|--------|
| 8.2.1 | Назначение | Генерация постов для Telegram-канала на основе исследований в интернете по заданному сценарию и по расписанию |
| 8.2.2 | Tools | Web search (Tavily/SerpAPI), RAG (общая или отдельная коллекция "контент канала"); шаблоны сценариев (например: "найти тренды за неделю в нише X и написать пост") |
| 8.2.3 | Сценарии | Модель `ChannelPostScenario`: name, prompt_template, search_query_template, schedule (cron); при срабатывании расписания — запуск агента → поиск → генерация текста поста |
| 8.2.4 | Публикация | После генерации: сохранение в черновики (модель `ChannelPost`: scenario, content, scheduled_at, published_at, status); отправка в канал через Bot API (метод для канала от имени бота или через связанный канал) |
| 8.2.5 | Очередь | Celery beat для расписания; задача: запуск агента → генерация → сохранение/отправка; настройка в кастомной админке |

**Результат Phase 8:** пользователи получают уведомления в Telegram; посты в канал генерируются AI по сценарию и времени и публикуются автоматически.

## Phase 9: Аналитика

**Цель:** сбор действий пользователей для маркетинга и партнёрской программы; отчёты и дашборды.

### Шаги

| # | Шаг | Детали |
|---|-----|--------|
| 9.1 | События | Модель `UserEvent` или логи в отдельную таблицу/сервис: user_id, event_type (page_view, button_click, purchase, signup, ref_click), payload (JSON), created_at; источник: frontend (отправка событий в API) и backend (события оплаты, регистрации) |
| 9.2 | API для событий | `POST /api/analytics/events/` — аутентифицированный или с анонимным id; валидация event_type по белому списку |
| 9.3 | Агрегаты для партнёрки | Уже частично в Phase 4; расширить: конверсия по ref, воронка (клик → регистрация → первая оплата) |
| 9.4 | Отчёты для маркетинга | Агрегации по когортам, по источникам (utm, ref); дашборд в кастомной админке: графики (по дням), таблицы; при необходимости экспорт в CSV |
| 9.5 | Хранение | Решение: одна БД с таблицей событий + материализованные представления/таблицы агрегатов по дням; или отдельный аналитический стек (ClickHouse, Metabase) — по масштабу проекта |

**Результат Phase 9:** события пишутся с фронта и бэка; партнёрская и маркетинговая аналитика доступна в админке и в ЛК партнёра.

## Зависимости и порядок выполнения

```
Phase 0 (инфраструктура) — обязательно первым
    ↓
Phase 1 (авторизация) — без неё нет пользователей
    ↓
Phase 2 (ядро, ЛК) — база для всего остального
    ↓
Phase 3 (контент) — можно параллельно с 4
Phase 4 (реферальная, партнёрская)
    ↓
Phase 5 (платежи) — использует заказы и партнёрку
    ↓
Phase 6 (бизнес-логика) — подключается к заказам и событиям
Phase 7 (AI-агенты, RAG) — можно после 2, зависит от 0
Phase 8 (уведомления, AI-посты) — после 1, 7
Phase 9 (аналитика) — после 2, 4, 5; события начинать рано
Phase 10 (админки) — после появления всех сущностей
```

**Рекомендуемая последовательность по спринтам:**

- **Спринт 1:** Phase 0 + Phase 1 + Phase 2 (каркас, вход, ЛК)
- **Спринт 2:** Phase 3 + Phase 4 (контент, рефералка, партнёрка)
- **Спринт 3:** Phase 5 + Phase 6 (платежи, плагины бизнес-логики)
- **Спринт 4:** Phase 7 (RAG + системный и персональный агенты)
- **Спринт 5:** Phase 8 + Phase 9 (уведомления, AI-посты, аналитика)
- **Спринт 6:** Phase 10 (обе админки, полировка)
