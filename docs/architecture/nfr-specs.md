# Non-Functional Requirements (NFR)

## Performance

| Требование | Целевое значение | Измерение |
|------------|-----------------|-----------|
| LCP (Largest Contentful Paint) | < 2.5 s | На типичном 3G (Lighthouse) |
| TTI (Time to Interactive) | < 4 s | На типичном 3G |
| API: время ответа (лендинг) | < 500 ms (p95) | GET /api/portfolio, /api/services и т.д. |
| API: отправка заказа | < 2 s (p95) | POST /api/orders/ до ответа клиенту |

**Меры:** Оптимизация сборки Vite (code splitting, минификация); сжатие статики (gzip/brotli); кэширование статики и при необходимости ответов API (Cache-Control); индексы БД по полям сортировки и фильтрации.

## Scalability

| Требование | Описание |
|------------|----------|
| Ожидаемая нагрузка (MVP) | До сотен визитов в день; единицы заказов в день |
| Масштабирование | Вертикальное (увеличение ресурсов VPS); при росте — горизонтальное (несколько инстансов Django за балансировщиком, отдельный сервис статики/CDN) |

На старте одного VPS достаточно.

## Availability

| Требование | Целевое значение |
|------------|------------------|
| Uptime (целевой) | 99% (допустимы плановые окна обслуживания) |
| Восстановление | Резервное копирование БД; документация по восстановлению на VPS |

## Security

| Требование | Описание |
|------------|----------|
| Транспорт | HTTPS везде (TLS 1.2+); сертификат Let's Encrypt |
| Секреты | SECRET_KEY, DB credentials, Telegram token — только в переменных окружения; не в коде и не в логах |
| Админка | Аутентификация (Django auth); сильные пароли; опционально 2FA |
| API | CORS только для домена лендинга; rate limiting на POST /api/orders/; валидация и санитизация входных данных |
| Заказы | Не логировать PII в открытом виде; доступ к заказам только авторизованному админу |

## Maintainability

| Требование | Описание |
|------------|----------|
| Код | Соответствие принятым конвенциям (Django, React); модульная структура (Django apps, компоненты React) |
| Документация | README, ARCHITECTURE, ADRs; описание деплоя и переменных окружения |
| Миграции | Django migrations для всех изменений схемы; не менять БД вручную в обход миграций |

## Observability

| Требование | Описание |
|------------|----------|
| Логирование | Логи приложения (Django, Gunicorn); ротация логов; не логировать пароли и токены |
| Мониторинг (опционально) | Health-check эндпоинт для Django; опционально — мониторинг uptime (внешний сервис) |
| Ошибки | При 5xx — логировать stack trace; не отдавать детали пользователю |
